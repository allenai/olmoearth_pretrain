## What was implemented

All three alternatives live as options in the existing `DualResEncoder`
(`olmoearth_pretrain/nn/dual_res_encoder.py`), selected by a new
`pixel_branch_type` config field. They share the existing plumbing (pixel patch
embed, ONLINE-unit packing, `PixelModalityState` interface to the pixel
reconstruction decoder and map probe, FSDP root-unit placement, per-step gradient
checkpointing), so the decoders and the model wrapper needed no changes. The original
branch is `pixel_branch_type="joint"` (the default) and its behavior is unchanged.

**1. `"conv"` — convolutional detail branch (`ConvPixelStep`)** (your top pick)
- Runs on the **dense** pixel grid: per (instance, timestep, band set) frame,
  all modalities concatenated on the frame axis.
- One step = a single DiT-style adaptively-modulated ConvNeXt unit:
  `frames += gate * MLP(dwconv3x3(norm(frames) * (1+scale) + shift))`, where
  (scale, shift, gate) come from each patch's coarse token (that's the coarse→pixel
  FiLM, broadcast over the patch's pixels through a 6D view — nothing materialized at
  pixel resolution).
- Pixel→coarse: mean-pool each patch's pixels → zero-init linear → residual add onto
  the coarse tokens (dense slab add; masked positions are dropped by the packing).
- Depthwise conv mixes across patch boundaries — the only variant with cross-patch
  pixel communication. To prevent masked-data leakage, **non-ONLINE pixels are zeroed
  before the first conv**; a regression test perturbs masked inputs and asserts all
  ONLINE outputs are bit-identical. (Caveat discovered along the way: at patch sizes
  other than `max_patch_size` the *coarse* branch's FlexiPatchEmbed bicubic-resizes
  the raw input, which already mixes data across patch boundaries — a pre-existing
  v1.2 behavior, not something the conv branch adds.)
- Extra knob: `pixel_conv_kernel` (default 3).

**2. `"window"` — patch-window attention with a coarse register (`WindowPixelStep`)**
- Window = exactly one unit's `P**2` pixels plus its coarse token projected down to
  the pixel width (+ a learned register embedding) as a `P**2+1`-th token: one batched
  `[num_units, P**2+1, Dp]` SDPA, **no padding and no mask** (every unit has exactly
  `P**2` pixels — much cheaper than the joint branch's padded per-location groups with
  sequences up to `max_units * P**2`).
- Bidirectional fusion for free: pixels read the register; the register's
  post-attention state is projected back up through a zero-init linear onto the
  coarse token.
- No cross-patch/temporal/modal pixel communication (coarse handles it).

**3. `"perceiver"` — cross-attention-only pixels (`PerceiverPixelStep`)**
- Pixels never mix with each other: per step, gated FiLM from the coarse token +
  a pointwise MLP. The pixel branch is a steered high-resolution memory; this is the
  ablation for whether fine-branch spatial mixing matters at all.
- Coarse←pixel read via a new `PixelReadPool`: the coarse token is projected *down*
  to the pixel width and used as a single query over its unit's pixels, with the
  pooled result projected back up (zero-init). Important: the first version reused
  the joint branch's `CrossAttnBlock` at coarse width — its `Dp→512` kv-projection of
  every pixel plus a 512-wide MLP made "the cheapest variant" the *slowest*; the
  pixel-width pool fixed that.
- Extra knob: `pixel_coarse_read_interval` (read every k-th pixel step; the last step
  always reads).

**Shared knob:** `pixel_every_k_blocks` — run the pixel branch (and both fusion
directions) only after every k-th coarse block (always including the last). This
turned out to be the main speed lever. Requires `depth % k == 0`.

All fusion projections are zero-initialized (AdaLN-Zero style), so every variant
starts exactly equal to the pixel-free coarse encoder while gradients still reach all
pixel parameters — verified by tests that backprop a coarse-only loss and check every
pixel parameter receives a gradient (this also keeps FSDP grad reduction
rank-uniform).

## Speed findings (the interesting part)

Baseline reality check: the original **joint branch measures ~8.7× a coarse-only
model** per training step (rank microbatch 32; it OOMs on a single 80GB H100 at the
production microbatch of 64, where coarse-only peaks at 35GB). Naively porting the
three alternatives got only 2.4–4.7×, and profiling showed the per-step *math* was
not the problem:

- **LayerNorm dominated everything** (~57% of conv_d32's CUDA time): under autocast
  LN runs in fp32, and its gamma/beta gradient reduction over millions of pixel rows
  (`GammaBetaBackward`) alone was ~28% of the step. Fix: affine-free
  (`elementwise_affine=False`) LayerNorms at pixel resolution in the new branch types
  — the affine is redundant before a Linear / FiLM modulation anyway (DiT does the
  same) — and the conv step was restructured to have a *single* pixel-resolution LN.
- **`index_copy` backward** for scattering coarse updates was ~7ms/block; replaced
  with `index_add` of the residual (identical result at unique indices, much cheaper
  backward).
- The **perceiver read at coarse width** (see above) — replaced with `PixelReadPool`.
- The **target encoder** (all-zero `token_exit_cfg`, skips all blocks) was still
  paying the dense pixel embedding of the unmasked batch every step — now skipped.
- Turning off pixel gradient checkpointing saves another ~0.3–0.5× but triples pixel
  memory (would OOM at microbatch 64) — left on.
- The pixel heads (reconstruction decoder + map probe) measure ~free (<0.05×).

After these fixes the per-step costs are bandwidth/overhead-bound and roughly equal
across the three designs (~0.11–0.2× of coarse-only per pixel step at heavy batch
mixes), so `pixel_every_k_blocks` sets the total.

## PixelDiT-style variant (added later on branch `favyen/20260710-pixeldit`)

A fourth design borrowing from **PixelDiT** (arXiv 2511.20645, NVlabs CVPR'26):
`pixel_branch_type="pixeldit"` in the same `DualResEncoder`. Structurally different
from the three interleaved variants — the pixel pathway runs **after the full coarse
trunk**, not inside it:

- **Post-trunk pathway**: the 12-block coarse trunk runs unmodified to completion;
  then `pixel_dit_depth` (M) tiny PiT blocks (`PixelDiTStep`) refine the pixel tokens
  at Dp=16–32.
- **Pixel-wise AdaLN** (the paper's distinctive conditioning): each unit's *final*
  coarse token projects to P² *distinct* per-offset (shift, scale, gate) sets — every
  pixel gets its own modulation, unlike the per-patch FiLM broadcast used by the
  other variants. The projection weight covers max_patch_size²=64 offsets and is
  row-gathered per actual patch size (flexi-safe; a P=1 batch computes 1/64th).
- **Token compaction attention**: each unit's P² pixel tokens compact via a learned
  per-offset flattening into ONE 512-d token; those attend over the trunk's packed
  per-instance sequence with the same attention mask and RoPE positions, then a
  learned per-offset expansion distributes the result back (gated residual). Every
  pixel gets a global receptive field at patch-sequence attention cost.
- **Fused output (our deviation from the paper)**: PixelDiT's pixel pathway directly
  emits the diffusion output; our losses/evals consume patch tokens, so the final
  pixel tokens are re-aggregated per unit by a zero-init learned compaction
  (`PixelDiTFusion`) and added residually to the output tokens. The latent-MIM loss
  and all in-loop evals therefore see pixel-refined tokens — unlike a strictly
  unidirectional pathway, downstream actually uses the pixel features. At init the
  model is exactly the coarse-only encoder (all gates and the fusion are zeroed).
- Pixel heads (reconstruction + map probe) consume the raw pixel features unchanged.

Measured (same protocol: 20 cases, seed 1, microbatch 64, vs coarse-only):

| launch script | config | typical mix | peak GB (mb64) | params |
|---|---|---|---|---|
| `pixeldit_d16_m4.py` | Dp=16, M=4 PiT blocks | **1.44×** | 38 | 92M |
| `pixeldit_d32_m2.py` | Dp=32, M=2 | **1.37×** | 39 | 90M |
| (also measured) d16_m2 | Dp=16, M=2 | 1.36× | 38 | 81M |

Params are higher than the other variants (the per-offset AdaLN tables are ~3M/block)
but activations stay small. Tests cover: fused-output gating (randomized pixel steps
cannot change the output while the fusion is zero; opening the fusion changes ONLINE
tokens only), pixel-wise (per-offset) AdaLN, step identity-at-init, and end-to-end
grad flow incl. the pixel heads.

## Cross-modality pixel reconstruction (fix, on `favyen/20260710-pixeldit`)

Investigating why W&B showed `pixel_loss/recon == 0` on many steps revealed the SSL
reconstruction loss was **structurally silent** in the launched runs:
`random_time_with_decode` assigns whole band sets an encode-only (ONLINE +
TARGET_ENCODER_ONLY, never DECODER) or decode-only (DECODER + TARGET, never ONLINE)
role per instance. With the v1.2 single-band-set S2/Landsat tokenization, a modality
therefore never contains both ONLINE and DECODER units — and the old
`PixelReconstructionDecoder` looked for context only within the same modality, so it
found zero reconstructable queries whenever ≥2 imagery band sets were present. The
occasional nonzero steps came from instances with exactly one imagery band set
present (single-band-set instances get mixed masking). `pixel_loss/map` was
unaffected (the map probe needs only ONLINE pixels + labels).

Fix: the reconstruction is now **cross-modality** — queries (every modality's DECODER
units, carrying a learned per-(modality, band set) embedding + timestep encoding)
attend to the ONLINE pixel reps of ALL pixel modalities at their location/offset
(shared spatial grid), implementing the decoder's originally documented intent.
Verified by a regression test that reproduces the production masking regime and
asserts a strictly positive loss with gradients flowing.

Cost of the now-active recon loss (it was ~free while silent): about +0.4× at the old
depth-2 setting, brought down by affine-free norms in its cross-attention blocks and
**recon depth 1** (now the launch-script default). Final same-process benchmark (20
cases, seed 1, microbatch 64): pixeldit_d16_m4 ≈ **1.55×** mean / 1.41× median,
pixeldit_d32_m2 ≈ **1.45×** / 1.29×, window_d64_e4 ≈ 1.6×, conv_d64_e4 ≈ 1.57× —
note run-to-run ratio noise of ±0.1 from coarse-baseline drift. A
`pixel_recon_location_ratio` knob exists to stochastically subsample reconstruction
locations (unbiased), but at 0.5 it bought nothing measurable: the residual cost is
dominated by data-dependent-shape ops (nonzero/unique/item syncs) in the grouping
path, a future optimization target.

Notes:
- The 8 launched runs predate this fix — their recon loss is mostly inactive (map
  probe carries the dense supervision), equally across variants, so comparisons
  between them stay fair. Relaunching with the fix would activate real SSL pixel
  supervision.
- The change is on `favyen/20260710-pixeldit` and cherry-picks cleanly onto
  `favyen/20260709-pixel-branch` if wanted there (the interleaved variants' launch
  scripts would also want `pixel_recon_depth=1` then).

## Caveats / notes for review

- The joint (`"joint"`) code path is untouched except for the mathematically
  equivalent `index_copy`→`index_add` swap in its coarse←pixel scatter.
- `PixelAttentionBlock`/`PixelFiLM` gained `norm_affine` flags that default to the
  old behavior; only the new branch types pass `False`.
- Benchmark ratios are conservative (single GPU, no optimizer/EMA/dataloader/comm;
  the loss and target forward are included). Two-view training doubles both sides
  equally.
- Batch-shape variance is real: worst single case (P=7, hw_p=8–9) is ~1.8–2.0× for
  the every4 configs; tiny cases carry ~1.5× of pure launch overhead.
- The conv branch computes on masked pixels too (dense layout, values zeroed) — ~2×
  the pixel positions of the packed variants; that's already priced into its numbers.
